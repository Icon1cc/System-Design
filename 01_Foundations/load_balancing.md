### Software Load Balancers

| Name | Type | Best For |
|------|------|----------|
| **NGINX** | L7 (and L4) | Web apps, reverse proxy |
| **HAProxy** | L4 and L7 | High performance, TCP/HTTP |
| **Envoy** | L7 | Microservices, service mesh |
| **Traefik** | L7 | Kubernetes, Docker |

### Cloud Load Balancers

| Provider | L4 Service | L7 Service |
|----------|------------|------------|
| **AWS** | NLB (Network LB) | ALB (Application LB) |
| **GCP** | TCP/UDP LB | HTTP(S) LB |
| **Azure** | Azure LB | Application Gateway |

## Interview Questions

### Question 1: "When would you use L4 vs L7 load balancing?"

**Strong Answer:**
```
L4 (Layer 4) - Use when:
â€¢ Need raw TCP/UDP load balancing (databases, game servers)
â€¢ Maximum performance is critical (lower latency)
â€¢ Protocol is not HTTP (custom protocols)
â€¢ Don't need content-based routing

L7 (Layer 7) - Use when:
â€¢ Need URL/path-based routing (/api, /static, /admin)
â€¢ Need SSL termination at load balancer
â€¢ Want HTTP-aware health checks
â€¢ Need to modify requests/responses
â€¢ Implementing microservices with different backends

Real-world: Most web applications use L7 because the
intelligence is worth the small latency cost. Use L4
for non-HTTP workloads or extreme performance needs.
```

### Question 2: "How do you make load balancers highly available?"

**Strong Answer:**
```
Load balancers can become a single point of failure. Solutions:

1. Active-Passive Pair
   â€¢ Two load balancers, one handles traffic
   â€¢ Standby monitors via heartbeat
   â€¢ On failure, standby claims Virtual IP (VIP)
   â€¢ Failover time: 1-30 seconds

2. Active-Active Pair
   â€¢ Both load balancers handle traffic
   â€¢ DNS or upstream LB distributes to both
   â€¢ Better utilization, instant failover
   â€¢ More complex configuration

3. Cloud-Managed Load Balancers
   â€¢ AWS ALB/NLB, GCP Load Balancer
   â€¢ Managed by cloud provider
   â€¢ Automatically highly available
   â€¢ Spans multiple availability zones

For critical production systems, I'd recommend:
â€¢ Cloud-managed LB for simplicity and built-in HA
â€¢ Or self-managed with Active-Active setup
```

## Common Pitfalls

### Pitfall 1: Forgetting about session persistence
If your app stores session on the server, users might lose their session when routed to a different server. Use sticky sessions or externalize session storage.

### Pitfall 2: Not monitoring load balancer health
The load balancer itself can become a bottleneck or fail. Monitor its CPU, memory, and connection counts.

### Pitfall 3: Aggressive health check settings
Too frequent health checks waste resources. Too infrequent means slow detection of failures. Balance is key (typically 10-30 second intervals).

### Pitfall 4: Equal weights for unequal servers
If servers have different capacities, use weighted algorithms to prevent overloading weaker servers.
